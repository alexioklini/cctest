Generic Sharing / Visibility Model

One mechanism — private · users · team · global — applied to chats, projects, scheduled tasks, workflows, and artifacts. Creator is always the owner; only the owner (or admin) sets visibility; ownership is transferable. Plan only — nothing implemented yet.

TL;DR

  • Projects already have the full model. project.json carries visibility (user|team|global) + owner_user_id + owner_team_id + extra_member_user_ids (individual grants) + excluded_user_ids. The work is to generalise that shape, not invent it.
  • Chats have a 2-state subset (visibility = user|team + team_id). No global, no individual grants, no explicit owner-vs-creator separation.
  • Schedules & workflows have nothing — only user_id (creator), owner-only access via _schedule_owner_check / RBAC scoping.
  • Artifacts have nothing — they inherit the visibility of the chat / project / scheduled-run that produced them, transitively. We'll keep that inheritance but make it explicit and let it be narrowed per-artifact (never broadened beyond the parent).
  • Favourites already model scope = user|team|general but only as a reference that mirrors the item's visibility — it never sets it. That stays true; the star button just won't let you favourite into a scope the item isn't visible in.
  • The big functional add behind finding 3 — a team document-ingestion pipeline (team__<tid> wing fed by folders, like project__<id>) — is folded in as Phase 6, optional.

1. The shared concept

Every shareable item gets the same five-field block (names chosen to match what project.json already uses, so projects need zero rename):

FieldTypeMeaning
owner_user_idstring (user id)The creator, set once at creation. Only this user (or an admin) can change visibility, edit the ACL, or transfer ownership. Never empty for items created after this lands; legacy rows may be empty → treated as "admin-owned".
visibilityprivate · users · team · globalThe base reach. private = owner + admin only. users = owner + the explicit grant list. team = owner + everyone in owner_team_id (+ optional extra individual grants). global = everyone except excluded_user_ids.
owner_team_idstring (team id) or ""Only meaningful when visibility = team. The uuid hex[:12] from user_teams.id.
extra_member_user_idslist[user id]Individual people granted access regardless of base visibility (the "share with specific users" capability). Additive — only ever widens. Meaningful for every visibility except global (where everyone already has it).
excluded_user_idslist[user id]Individual people denied access. Only meaningful when visibility = global (an allow-list everywhere else makes exclusions moot). The owner is never excludable from their own item.
Naming note: projects currently use visibility = "user" for "owner-only". We rename that value to "private" in the vocabulary going forward, but keep a read-time alias ("user" → "private") so existing project.json files don't need rewriting. The new "users" value (plural) is the "specific users" tier — distinct from "private". Yes, "user" vs "users" is a confusing one-letter difference in legacy data; the migration normalises legacy "user"→"private" lazily on next write.

2. Where each item type stands today

2.1 Projects — complete

brain.py → ProjectManager.list_projects / create_project / update_project already filter on the five-field block; server_lib/auth.py → can_access_project / can_manage_project already encode "owner or admin manages, members read". Gap: no owner_user_id backfill for legacy projects (some project.json have empty owner), and the value "user" needs aliasing to "private". can_manage_project is already strictly owner-or-admin — that's the model we want everywhere.

2.2 Chats — partial

Has
sessions.user_id (creator), sessions.visibility ∈ {user, team}, sessions.team_id. list_sessions(visible_user_ids, visible_team_ids) already does UNION-style scoping; handlers/projects.py → _session_access_check already does owner / team-member / team-head-manages.
Missing
global tier; users tier (individual grants); explicit owner_user_id separate from user_id (they're the same today — fine, but the share dialog talks about "owner", so we expose user_id under that name and add transfer); excluded_user_ids.
Decision
Don't add an owner_user_id column — reuse user_id as the owner. Transfer = UPDATE user_id (+ re-point the MemPalace wing, see §5). Add the three JSON columns + extend visibility domain.

2.3 Scheduled tasks — none

schedules.user_id = creator. _schedule_owner_check(name) gates every mutating op to creator-or-admin; non-admins see only their own schedules in the list. Legacy schedules with empty user_id are admin-only. No sharing dimension at all. Adding the five-field block here is the biggest net-new surface — both the task definition (who can see/edit/run it) and the runs (whose artifacts land where, who can view the run history).

2.4 Workflows — none (definition) / user-scoped (runs)

Workflow definitions are bare .workflow source files in agents/<a>/workflows/ with no metadata at all — no owner, no visibility. Any authenticated user who can reach the agent can list/run them. Workflow runs (workflow_history rows) carry user_id and the v8.23.10 delete helpers already scope non-admins to their own runs. So: definitions need a metadata sidecar (or table); runs need the denormalised visibility snapshot.

2.5 Artifacts — inherits, implicitly

An artifact is visible to whoever can see the session / project / scheduled-run that wrote it (the artifact folder lives inside that parent's tree, and listing artifacts goes through the parent). There's no per-artifact control. We keep inheritance as the default and add an optional visibility_override that can only narrow. (Use case: a scheduled task is team-visible, but one run produced a draft the owner wants to keep private until polished.)

2.6 Favourites — already scoped, stays a mirror

favourites table has scope ∈ {user, team, general} + scope_id. Its docstring is explicit: a favourite "never broadens visibility, only references it." That stays true. The only change: the star button's scope options are filtered to scopes the item is actually visible in (you can't pin a private chat into the general favourites bar). When an item's visibility is narrowed and orphans a favourite, a cleanup pass removes the now-invalid favourite rows (or marks them stale — see §10 Q5).

3. Access & manage rules — one function family

Today there are bespoke checks: can_access_project, _session_access_check, _schedule_owner_check, can_delete_resource, the workflow-history RBAC scoping. We collapse the logic into two pure functions in server_lib/auth.py that take the five-field block as a dict, plus thin per-item adapters that load the block and call them.

# server_lib/auth.py  (new — generalises can_access_project)

VISIBILITY_VALUES = ("private", "users", "team", "global")
_LEGACY_VIS_ALIAS = {"user": "private"}   # project.json migration

def normalize_visibility(v: str) -> str:
    return _LEGACY_VIS_ALIAS.get(v or "private", v or "private")

def can_access(user: dict, block: dict) -> bool:
    """block = {owner_user_id, visibility, owner_team_id,
                extra_member_user_ids, excluded_user_ids}"""
    if user["role"] == "admin" or user["id"] == "__system__":
        return True
    owner = block.get("owner_user_id") or ""
    # Legacy: no owner → admin-only (already handled above) — but be permissive
    # for chats that pre-date ownership: caller decides via `legacy_open=True`.
    if owner and owner == user["id"]:
        return True
    vis = normalize_visibility(block.get("visibility"))
    extras = block.get("extra_member_user_ids") or []
    if user["id"] in extras:
        return True
    if vis == "global":
        return user["id"] not in (block.get("excluded_user_ids") or [])
    if vis == "private":
        return False                       # owner + extras only (handled above)
    if vis == "users":
        return False                       # owner + extras only (handled above)
    if vis == "team":
        tid = block.get("owner_team_id") or ""
        if not tid:
            return False
        return any(t["id"] == tid for t in AuthDB.get_user_teams(user["id"]))
    return False

def can_manage(user: dict, block: dict) -> bool:
    """Edit metadata, change visibility, edit ACL, transfer, delete.
    Strictly owner-or-admin — NO team-head shortcut for non-project items
    unless we decide otherwise (see Open Q2)."""
    if user["role"] == "admin" or user["id"] == "__system__":
        return True
    owner = block.get("owner_user_id") or ""
    return bool(owner) and owner == user["id"]

Per-item adapters (each ~5 lines): project_block(project_cfg), session_block(session_info) (maps user_id→owner_user_id), schedule_block(sched_row), workflow_block(meta), artifact_block(artifact_row, parent_block) (resolves the override, clamps to ≤ parent reach). can_access_project / can_manage_project become two-line wrappers so nothing downstream breaks.

"Creator is owner" is enforced at the create path, not the access path. Every create_* function stamps owner_user_id = _thread_local.current_user_id and ignores any owner passed in the body (admins excepted, for seeding). The access functions just read it.

3.1 List-scoping queries

Every "list my items" endpoint already takes (or will take) visible_user_ids + visible_team_ids derived from the caller. The SQL/filter changes:

  • Add the global arm: OR visibility = 'global' (then post-filter out rows where caller ∈ excluded_user_ids — cheap, JSON parse on the handful of global rows).
  • Add the users arm: can't do this purely in SQL with a JSON column portably, so: fetch candidate rows (own + team + global), then in Python keep any row where caller_id ∈ extra_member_user_ids. For chats this is list_sessions in server_lib/db.py; for schedules a new filter in the schedules query; etc.
  • Legacy rows (empty owner): chats → visible to all authenticated (current behaviour, keep); schedules → admin-only (current behaviour, keep); workflows → all authenticated (current behaviour, keep) until an owner adopts them.

4. Ownership transfer

One endpoint shape, parameterised by item type: POST /v1/share/transfer with body {item_type, item_id, agent_id?, new_owner_user_id}. (Or per-resource: POST /v1/agents/{a}/projects/{name}/transfer etc. — see §10 Q1.) Rules:

  • Caller must pass can_manage on the current block (owner or admin).
  • new_owner_user_id must exist in auth.db and have access to the agent (for agent-scoped items).
  • On success: set owner_user_id = new_owner_user_id; remove the new owner from extra_member_user_ids / excluded_user_ids if present; do not change visibility (the new owner can narrow it afterward); write an audit-log row (action='ownership_transfer', old/new owner, item ref).
  • MemPalace consequence (chats/projects only): if the item's wing is keyed by owner (user__<uid>), the existing drawers stay where they are (re-keying drawers is expensive and the old owner already saw that content) — but future sync goes to the new owner's wing. Document this asymmetry. Alternative for projects: project wings are keyed by project id, not owner — so projects have no wing churn on transfer. (This is one more reason to keep chat wings as-is on transfer rather than re-point.)
  • Schedules: transfer also moves which user's quota the runs bill against going forward (_log_call_cost reads current_user_id — the scheduled-run worker sets it from schedules.user_id). Note this in the transfer confirm dialog.

5. MemPalace wing routing under the new model

_resolve_session_wing(info) in server_lib/db.py currently: project → team → user → "". Under the extended visibility domain it becomes:

Effective visibility of the chatWing the chat-sync daemon files into
project-scoped (session has a project)project_chat__<project_id> — unchanged, takes precedence over everything below
teamteam__<owner_team_id> — unchanged
private / usersuser__<owner_user_id> — same as today's user; the users grant list does NOT get its own wing (the granted users can read the chat via the session, but its memory stays in the owner's private wing — granting chat access ≠ granting memory access). If we later want shared memory for a users-scoped chat, that's a separate feature.
globalOpen question (Q4). Options: (a) still user__<owner> — memory stays private even though the chat is world-readable; (b) a new shared__global wing that everyone reads cross-wing (like the untyped brain_code wing). Leaning (a) for v1 — least surprising, no new wing, and "global chat" is rare.

Scheduled-run artifacts: today they go to <agent_id>_artifacts (shared per-agent). Under the new model, if the schedule is private we'd arguably want them in user__<owner> instead — but that's a behaviour change to the miner with its own blast radius; defer to Phase 6 / a follow-up and keep <agent>_artifacts for now (it's already a shared/untyped wing, so it's the most permissive — narrowing it later is safe, widening would be the dangerous direction).

6. GUI — the share dialog

One reusable component — shareDialog(itemType, itemId, agentId) in web/js/share.js — opened from a "Share" / people-icon button that appears in:

  • Chat: in the chat header, next to the existing star (favourite) and title-edit controls.
  • Project: in the project detail right-pane (replaces / absorbs the current admin-only visibility selector).
  • Scheduled task: in the schedule edit modal.
  • Workflow: in the workflow list row actions + the workflow editor toolbar.

9. Edge cases & invariants

  • Owner is never excludable. can_access short-circuits on owner before consulting excluded_user_ids. The dialog refuses to add the owner to the exclude list.
  • Creator = owner, always, at creation. Every create_* ignores an owner field from the request body (admin seeding aside). No path lets a non-owner end up owning something they didn't create except via explicit transfer.
  • Only owner/admin changes visibility or ACL. Enforced by routing every PATCH …/share + transfer through can_manage. Team members of a team-visible item can read it but not re-share it (matches today's project behaviour where visibility is admin-only).
  • Grants/excludes don't survive a tier change that makes them meaningless — switching to global drops extra_member_user_ids (everyone's already in); switching away from global drops excluded_user_ids (no allow-list to subtract from). The handler normalises on save (mirrors handlers/projects.py which already does this for excluded_user_ids).
  • Artifact override can only narrow. Validated on save; an override wider than (or incomparable-and-not-a-subset of) the parent's effective visibility → 400.
  • Transfer doesn't re-key existing MemPalace drawers — and that's the deliberate choice, not a TODO. A chat's drawers live in the wing user__<owner_uid>, where the uid is baked into the Chroma wing field on every drawer + closet row and into the per-wing closet index — there's no column to flip; re-keying means a bulk delete-from-old-wing / insert-into-new-wing + a closet rebuild on both wings, racing the chat-sync daemon, with all the half-migrated-segment / orphaned-HNSW failure modes the miner already has cleanup code for. It also buys little: the old owner already read every drawer (transfer doesn't un-ring that), and the new owner gets all future drawers anyway (_resolve_session_wing returns the new owner's wing post-transfer). Projects are unaffected entirely — project wings are keyed by project_id (stable uuid), not by owner. Decision: never auto re-key. The transfer confirm dialog states the asymmetry in plain language. If offboarding ever needs the back-catalogue moved, that's an opt-in "migrate this chat's memory too" checkbox firing a single-session re-sync (closet rebuild contained to the two wings) — listed as Open Q, not built.
  • After transfer, the previous owner can still surface this chat's content via mempalace_query even if the new owner later narrows visibility to private — because memory drawers aren't re-ACL'd on transfer (they sit in the old owner's wing, see above). This is an accepted minor leak: the old owner had full read access for the chat's entire life, so nothing new is exposed; it's the same data they already saw, just reachable via memory search rather than the chat list. The one scenario where this matters — offboarding, where the old owner's whole user__ wing is being wiped — is exactly the case the opt-in migrate-and-purge path above is for.
  • Legacy rows keep their current effective behaviour until adopted: anonymous/owner-less chats → all-authenticated read; owner-less schedules → admin-only; owner-less workflows → all-authenticated. The first can_manage-passing edit by a non-admin claims ownership (workflows) or is admin-only (schedules) — see Open Q6.
  • Favourites never widen. Star scope options filtered to the item's visible scopes; orphaned favourites cleaned up (or marked stale) when an item is narrowed.
  • Scheduled-run billing follows the schedule's current owner. The run worker reads schedules.user_id for current_user_id; after transfer, new runs bill the new owner's quota. (Past costs.db rows are immutable.)
  • KV-cache stability: none of this touches _build_system_prompt's static text — visibility is metadata, not prompt content. The per-session preamble may mention "shared with team X" but that's already in the dynamic preamble bucket, not the cached prefix.

10. Open questions (need your call before / during implementation)

  1. One /v1/share/* endpoint family vs per-resource endpoints? Generic is DRYer for the client; per-resource matches the existing routing style (/v1/agents/{a}/projects/{name}/…). Leaning: a generic share.js client that calls per-resource endpoints (server stays conventional, client stays DRY).
  2. Does can_manage get a team-head shortcut for non-project items? Today projects: no (owner-or-admin only). Schedules/chats/workflows: undecided. A team head arguably should be able to manage team-visible items their members created. Safe default: no shortcut (matches projects); revisit if it's annoying. The one exception that's clearly right: team-head manages the team's input-folder list in Phase 6.
  3. Workflow metadata: sidecar file vs DB table? Sidecar keeps the workflow self-contained and mirrors project.json; table is easier to query for "list all workflows visible to X" without stat-ing every file. Leaning sidecar (workflow counts are small).
  4. Where does a global chat's memory go? v1: stays in user__<owner> (least surprising, no new wing). Alternative: a shared__global wing. Decide before P1's _resolve_session_wing change.
  5. Orphaned favourites on narrowing: delete or mark stale? Delete is cleaner; mark-stale lets the user re-pin if they regain access. Leaning delete + a toast.
  6. Adopting legacy owner-less items: for workflows/chats, does the first non-admin can_manage edit claim ownership, or is adoption admin-only? Leaning: first-edit-claims for workflows & chats; admin-only for schedules (they bill quota).
  7. Do schedule_history / workflow_history rows need a denormalised visibility snapshot, or is it fine to always resolve through the live parent? Snapshot = old runs don't get re-exposed if the parent is later widened, but more columns + a subtle "why is this run still private" UX. Live = simpler, but widening a workflow retro-exposes its run history. Leaning: snapshot for runs (the safer default for "I made this private after the fact").
