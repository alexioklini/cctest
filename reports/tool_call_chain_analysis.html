1. TL;DR — Why the same scheduled task produces 192 chars one day and 30 KB the next

Three independent forces compound:

  1. The engine has no “did the model promise but not deliver?” check. When Mistral Medium 3.5 emits “I'll now write the report. Let me compile…” (32 tokens, finish_reason="stop", no tool_calls), _handle_openai_response returns that string as the final answer. §10
  2. Tool-result shape mutates non-deterministically across rounds. The middleware _microcompact and _compress_old_tool_results rewrite older tool results in place every 2 rounds / when over budget; the worker subagent path summarises via an LLM; auto tools flip between raw passthrough and isolated worker envelope at the 8 KB boundary. Same prompt → different in-context bytes → different sampling. §6
  3. The storage cap that lopped run 786 mid-sentence. complete_execution was hard-truncating schedule_history.result at 10 000 chars (brain.py:15590). Fixed in this PR — full delegate output is now stored verbatim.
The 32-token failure mode The engine treats finish_reason=stop + empty tool_calls_map + non-empty full_text as a successful completion regardless of what the text says. A model can announce its intention to call a tool, then stop, and the loop has no way to recover. The only mitigations available without code change are: (a) lower temperature, (b) raise the per-task max_tool_rounds so even sloppy turns get more chances, (c) phrase the task so the model writes inline instead of via a file. A real fix needs a server-side intent-vs-action guard.

2. The engine — one round, end to end

The “agentic loop” is not a loop. It is recursion through send_message: each round is one send_message call, the SSE consumer dispatches tools and recursively calls send_message with _tool_round + 1. Python stack depth = round count.

Mutation in place _microcompact and _compress_old rewrite the messages list in place. The next round sees a context that is structurally different from what the previous round emitted. This is one of the strongest sources of run-to-run variance for the same prompt.

3. Parsing the model's response into tool calls

3.1 Streaming aggregation

tool_calls_map = {}                                  # brain.py:25876
for tc in (delta.get("tool_calls") or []):           # brain.py:26005 — guards Gemini's null
    idx = tc.get("index", 0)
    tool_calls_map.setdefault(idx, {"id": "", "name": "", "arguments": ""})
    if tc.get("id"):                                  # may arrive on a later delta
        tool_calls_map[idx]["id"] = tc["id"]
    if tc.get("function", {}).get("name"):
        tool_calls_map[idx]["name"] = tc["function"]["name"]
    tool_calls_map[idx]["arguments"] += tc.get("function", {}).get("arguments", "")

Keyed by provider index so out-of-order shards reassemble. arguments concatenated as raw strings — never parsed during streaming.

3.2 Gemma fallback (text-mode tool calls)

When tool_calls_map is empty and the assistant text contains <|tool_call>, _parse_gemma_tool_calls regexes them out (brain.py:26074-26084). Generates synthetic ids gemma_<8 hex>. This is the only text-based tool-call parser. An OpenAI-shape JSON call buried in a Mistral text block is NOT parsed.

3.3 Malformed JSON arguments

try:
    args = json.loads(tc["function"]["arguments"])
except json.JSONDecodeError:
    args = {}                                         # silently coerced — call still runs

Tools that receive {} typically return their own parameter-validation error; that error becomes the next-round tool result.

4.3 Parallel batch dispatch

_execute_tools_batch partitions the call list into consecutive runs of same class:

  • Consecutive concurrent-safe tools ≥ 2 → ThreadPoolExecutor(max_workers=min(5, len(batch))).
  • Anything else → sequential, in arrival order.

Concurrent-safe set (brain.py:23641-23649): read_file, list_directory, search_files, read_document, memory_recall, memory_shared, exa_search, web_fetch, code_graph_query, schedule_list, schedule_history, list_nodes, task_status, context_search, context_detail, context_recall, git_command (git further restricted to read-only subcommands at brain.py:23680-23683).

Thread-locals are snapshotted from the parent (_parent_ctx in _execute_tool_in_thread, brain.py:23758) and re-installed in each worker thread: current_session_id, current_agent, mcp_manager, current_user_id. Without this, session-scoped dedup and MCP routing silently fork.

Order is preserved. Results collected via as_completed into a result_map, then re-appended in original batch order (brain.py:23735-23737).

No batch-level timeout. Each tool body owns its own timeout. DEFAULT_PROFILES.timeout_seconds is currently advisory metadata only.

Per-tool exceptions are caught and wrapped: result = _err(f"Tool execution error: {e}") (brain.py:23728-23729) — they never abort the batch.

5. How the caller (the loop) gets the result back

This is the dimension that matters most for your variance question: the shape of what reaches the model on the next round can differ wildly even for the same tool. There are four distinct return shapes.

5.1 The two-step return contract

The "caller" here means the agentic loop in _handle_openai_response. Every tool body must return a string. The path from string to "what the model sees on next round" is:

# brain.py:23728
result = future.result()           # may raise → caught → result = _err(...)
result_map[tc["id"]] = result      # keyed by call id
event_callback("tool_result", {"name": tc["name"], "result": result, "tool_round": N})

# brain.py:23736-23737
for tc in batch:
    results.append({"tool_use_id": tc["id"], "result": result_map[tc["id"]]})

# brain.py:26228-26240
for br in batch_results:
    sanitized = _sanitize_tool_result(name, br["result"])     # 30 KB cap, base64 strip
    sanitized = _extract_and_save_mcp_blobs_direct(sanitized, session_id)
    messages.append({"role": "tool", "tool_call_id": br["tool_use_id"], "content": sanitized})

From this point, the middleware chain (_run_middleware) may further mutate the appended tool messages before the recursive send_message call.

5.3 The event_callback("tool_result", …) side channel

Independent of what reaches the model: every tool result is also emitted on the SSE bus (brain.py:23733, 23753). This is what:

  • Persists the tool call to the chat DB via handlers/chat.py:760-768 with its own per-tool caps (50 000 chars for read tools, 5 000 elsewhere — into msg_metadata, not the user-visible message body).
  • Drives the web UI live "Tool called: X" badges.
  • For scheduled tasks, increments tool_calls counter persisted to schedule_history.tool_calls.

6. Post-result processing — sanitize, summarise, isolate, compact

6.1 The summariser pipeline (heavy + auto-isolated tools)

_summarise_tool_result (execution.py:596-694). When invoked:

  • Heavy path → ALWAYS, inside run_worker_subagent at execution.py:890.
  • Auto path → only when post-execution size > auto_threshold_bytes = 8192 (execution.py:951, 969).
  • Light path → NEVER.

Mechanics:

  1. Resolve summariser model. Priority: agent.config["summariser_model"] → _thread_local._current_model (the chat's own model). No model → returns _static_summary immediately.
  2. Truncate raw to summariser_max_input_chars = 32 000.
  3. gdpr_pick_model_for_background may swap to a local fallback; GDPRBlockedError → static summary.
  4. send_message_with_fallback(messages, model, …, tools=False, inference_params={"max_tokens": 512, "temperature": 0.0}). Recursive into the engine, but tools=False means no nested tool loop.
  5. _parse_summariser_output splits the response at "SECTIONS:" → (summary, sections).
  6. Returns 3-tuple (summary, sections, usage_dict).
Historic bug — the 3-tuple unpack maybe_retroactive_isolate at one point unpacked 2 values from _summarise_tool_result (which returns 3), causing every read_document > 65 KB to crash with ValueError: too many values to unpack. The crash was caught and rebranded as a tool error, surfacing as hallucination. Live callers now correctly unpack 3 (execution.py:890, execution.py:969) — listed in memory note bug_summarise_tool_result_unpack.md. Don't break this.

9. Sources of non-determinism

Same scheduled task, same model, range 192 → 30 000 chars. Contributors, roughly ranked by impact:

#SourceWhy it bites
1Sampling temperature 0.7Provider RNG seed not pinned. A coin flip at the comma after "Let me…" decides between calling write_file and stopping.
2Auto-isolation 8 KB boundarySame URL fetched twice can produce 7 944 bytes one day (raw passthrough) and 8 213 the next (worker envelope w/ summary). The model sees entirely different inputs.
3_microcompact every even roundRewrites older tool messages in place. Drops more on a 6-round run than a 4-round run.
4Tool-result summariser uses session modelThe summariser runs on the same chat model. If a fallback or GDPR swap kicks in, the summary text is different — and that summary text is what reaches the model next round.
5GDPR auto-fallback model swapPer-request PII detection; matches depend on exact prompt text. A web_fetch result that mentions a real-looking phone number swaps the next model to local.

10. Failure modes & the "I'll write it…" stall

10.1 Termination outcomes for a scheduled run's stored result

TerminationWhat goes into schedule_history.result
Normal completionThe model's final full_text from the last _handle_openai_response call.
1.5× hard cap hitThe literal string "[Tool round limit reached — stopping to prevent runaway loop. Check chat for partial progress.]". All prior round text is lost. Run 778 was this. Look in the artifact folder for what was actually produced.
Max-output recovery exhaustedfull_text + "\n\n⚠️ *Output token limit reached…*".
finish_reason="stop" mid-thoughtThe model's truncated text returned as-is. Run 779 and 784 were this — 32-token "Let me compile…" with no follow-up tool call. No detection, no recovery, the engine treats this as success.
Soft cap → tools=False retryOne more LLM call without tool defs; whatever text it produces.
Reactive 400 + compactionIf recovery succeeds: retried call's text. If both microcompact and full compact fail: send_message returns None; fallback chain.

10.2 The 32-token failure — exact code path

  1. SSE stream ends. full_text = "I'll now write the report. Let me compile…". tool_calls_map = {}. finish_reason = "stop".
  2. Truncated-tool discard (brain.py:26108) — skipped, no tool calls.
  3. Length recovery (brain.py:26122) — skipped, finish_reason is "stop" not "length".
  4. if not tool_calls_map: branch at brain.py:26173-26184 → end trace span → reset recovery counter → return full_text.
  5. send_message returns the 32-token string. _run_delegate returns it. _execute_scheduled passes it to complete_execution. Done.
No intent-vs-action guard exists There is no code anywhere in _handle_openai_response / send_message / _run_delegate that inspects assistant text for unfulfilled promises ("I'll create…", "Let me write…", "Now I'll save…"). finish_reason="stop" + non-empty text = success, no matter what the text says. The same path triggers if the provider emits finish_reason="tool_calls" but the actual tool_calls deltas were dropped (provider bug, proxy stripping, SDK quirk) — empty tool_calls_map means the finish_reason string itself is never inspected past the two == "length" checks.

10.3 Practical mitigations (no engine change)

  • Lower temperature. For Mistral Medium 3.5 on policy-reproduction the memory note flags 0.2 / top_p 0.85 as the validated settings. Default 0.7 is the worst offender.
  • Phrase the task to write inline. "Produce the report directly in this reply" beats "Write the report to a file" for a single-shot scheduled task — eliminates the write_file transition where the stall lives.
  • Raise per-task max_tool_rounds so soft cap fires at a higher round; gives more chances to recover from a transient stop.
  • Reduce tool fan-out before synthesis. Each tool-result message is more context for the model to lose its place in. Aim for < 6 rounds before the synthesis turn.

10.4 Engine-level fix candidates (would need code change)

  • After finish_reason="stop" + empty tool_calls_map, regex-check full_text for unfulfilled-intent markers ("I'll (now |)?(write|create|save|compile)" at the tail) and re-prompt once.
  • Detect finish_reason="tool_calls" with empty tool_calls_map as a provider bug → re-prompt with "your tool call was not received".
  • Add a server-side completion sanity check: if the last assistant text mentions a file path that doesn't exist on disk and was supposed to be created, re-prompt.
