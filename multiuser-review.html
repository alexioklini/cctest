0. Executive summary

Brain has a complete, layered multi-tenant model bolted on top of what was originally a single-user agent server. Identity lives in a dedicated SQLite database (agents/main/auth.db); authorization is enforced both by path-based gates in the HTTP layer and by per-resource ownership checks in handlers. Memory is isolated by namespacing every drawer into a "wing" derived from the session's owner / team / project.

LayerMechanismMaturity
AuthenticationStateless JWT (HS256, 24 h default), bcrypt password hashes, optional self-registration (default off)solid
Authorization — coarse3 roles (admin / poweruser / user) + 4 capability flags; path allowlists for admin-only endpointssolid

Key files referenced throughout: server_lib/auth.py (1110 LOC — AuthDB, JWT, authz helpers), handlers/auth.py (681 LOC — auth/user/team endpoints), server.py (auth middleware, path gates), server_lib/db.py (session schema, _resolve_session_wing, chat-sync cursor), handlers/projects.py & handlers/sessions_handler.py (resource access checks), brain.py (ProjectManager, tool_mempalace_query, user-profile mirroring).

1. User management

1.1 Storage

Users live in agents/main/auth.db, managed by the AuthDB class in server_lib/auth.py. The users table schema:

users(
  id            TEXT PRIMARY KEY,   -- uuid4 hex[:12]
  username      TEXT UNIQUE NOT NULL,
  password_hash TEXT,               -- bcrypt
  display_name  TEXT DEFAULT '',
  email         TEXT DEFAULT '',
  role          TEXT DEFAULT 'user',-- 'admin' | 'poweruser' | 'user'
  created_at    REAL,
  last_login    REAL,
  disabled      INTEGER DEFAULT 0,
  capabilities  TEXT,               -- JSON: per-user feature-flag overrides
  preferences   TEXT                -- JSON: user-controlled settings (see §1.3)
)

1.2 Lifecycle

Create
AuthDB.create_user(username, password, role="", display_name="", email="") — mints a 12-char hex id, bcrypt-hashes the password, applies default_role() if role is blank, and auto-grants the new user access to the configured default agents (default ["main"]).
First-run admin
If the table is empty on init, a default admin / admin user (role admin) is created and a "change immediately" warning is printed to the server log.
Admin update
update_user(user_id, updates) — admin-only; can change display_name, email, role, disabled, capabilities. Does not touch preferences.
Self-service
update_self_profile (display name / email), update_preferences (see §1.3), change_password (requires old password).
Admin password reset
admin_reset_password(user_id, new_password) — no old password required.
Disable
disabled=1 — token verification rejects the user on next request (no session revocation table needed since JWT is stateless, but refresh_token and verify_token both re-load the user and bail on disabled).

1.3 Per-user preferences (users.preferences JSON)

Validated against PREFERENCE_DEFAULTS via _coerce_pref; default-valued keys are pruned on write. Surfaced to the agent in the round-0 first-user-message preamble (kept out of the system prompt to preserve warm-pool KV-prefix stability).

1.4 Capabilities (users.capabilities JSON)

Sparse per-user overrides on top of role defaults (CAPABILITY_DEFAULTS). Admin bypasses every capability check.

1.5 Self-registration

POST /v1/auth/register exists but is disabled by default — gated by registration_enabled() ⇐ config.auth.registration_enabled. When enabled it creates a user with default_role() and returns a token immediately. The server prints Auth: enabled (registration: open|closed) at startup so the operator knows the posture.

2. Teams & team heads

2.1 Schema

Two tables in auth.db:

user_teams(
  id           TEXT PRIMARY KEY,    -- uuid4 hex[:12]  → this is the MemPalace team wing key
  name         TEXT UNIQUE NOT NULL,
  description  TEXT DEFAULT '',
  head_user_id TEXT NOT NULL,       -- FK users.id
  created_at   REAL
)
user_team_members(
  user_id  TEXT NOT NULL,           -- FK users.id
  team_id  TEXT NOT NULL,           -- FK user_teams.id
  joined_at REAL DEFAULT (now),
  PRIMARY KEY (user_id, team_id)
)  -- indexed on user_id and team_id

2.2 Team head — what makes it special

AspectBehaviour
Who can be headcan_create_team() requires the creator to be poweruser or admin; the head is the creator.
Auto-membershipThe head is inserted into user_team_members at team creation.
Cannot be removedremove_team_member() refuses to remove the head.
Management rightscan_manage_team(user, team_id) ⇒ admin OR the team's head_user_id. Gates add/remove member, update team, delete/dissolve team.
Cross-user visibilityget_visible_user_ids(user) returns [user_id] + members-of-every-team-where-user-is-head. So a team head can see and read the chats & projects of all their team members. Regular members do not get this — they only see their own.
Session/project management of team resourcesA session/project with visibility='team' can be read by any team member but managed (deleted, settings changed) only by the head (or admin, or original owner).

2.3 Relationship to agent teams

There are two distinct "team" concepts in this codebase — don't conflate them:
• User teams (this section) — groups of human users in auth.db, with a head_user_id. Drive RBAC visibility and the team__<tid> memory wing.
• Agent teams — the team field in agents/<name>/agent.json + team.members. This is about which agents can delegate to which other agents (orchestration topology: main → heads + standalone; heads → their members; members → peers + head). Unrelated to human users.

2.4 Team-scoped resources

Sessions
sessions table has team_id TEXT DEFAULT '' and visibility TEXT DEFAULT 'user'. A team chat is visibility='team' + team_id set.
Projects
project.json carries owner_user_id, owner_team_id, visibility (user/team/global), plus exception lists (extra_member_user_ids, excluded_user_ids). Creating a team-visible project requires the caller to be that team's head.
Agent / model ACLs
user_agent_permissions, team_agent_permissions, user_model_permissions, team_model_permissions — grant tables. Effective access = direct grant ∪ any-team grant ∪ admin.

3. RBAC & capability gating

3.1 Two enforcement points

① Path-based gate — BrainAgentHandler._auth_gate(path, public_paths, admin_paths, method) in server.py. Called at the top of do_GET / do_POST / do_PUT / do_DELETE. Logic:
  1. If path ∈ public_paths → return current user (or SYNTHETIC_ADMIN if auth disabled), no auth required.
  2. Else _require_auth() — 401 if no valid token.
  3. If the path is admin-only (via per-method _is_admin_get/post/delete() prefix/exact matching OR explicit admin_paths set) and user.role != 'admin' and user.id != '__system__' → 403 "Admin access required".
  4. If the path requires a capability (_path_requires_capability) and the user lacks it → 403 "Capability 'X' not granted".
② Per-resource ownership check — invoked inside handlers once the path gate passes:
  • _session_access_check(sid, require_manage=…) — admin/__system__ ⇒ ok; owner ⇒ ok; legacy ownerless session ⇒ ok (anyone authenticated); visibility='team' + caller in team ⇒ read ok, manage only if caller is head; otherwise 403.
  • _project_access_check(agent, proj, require_manage=…) ⇒ can_access_project() / can_manage_project().

3.2 Roles

RoleCan doCannot do
adminEverything. Bypasses every capability check and every per-resource ownership check. Manages users, teams, providers/models, agents (hooks/commands/workflows), permissions, sees all chats & projects, reads audit log, backups.—
poweruserChat; create & manage own projects; create user-teams & act as team head; see/read team members' resources (when head).Server/agent config, provider/model management, workflows, skills install, user management, audit log.
userChat with permitted agents/models; create artifacts; edit own profile/preferences/password; join teams (when added by a head).Create projects (default), workflows, skills install, anything admin, see other users' resources.

3.3 Admin-only endpoints (representative)

Source of truth: _is_admin_get/post/delete() + the _ADMIN_*_PATHS sets in server.py. Highlights:

  • GET: /v1/auth/users, /v1/auth/audit, /v1/providers, /v1/mempalace/* (memory stats / classifier config), /v1/agents/* config endpoints.
  • POST: /v1/auth/users (create/update/disable/reset-password), /v1/auth/permissions (grant/revoke agent & model access), /v1/auth/migrate, /v1/agents/create, /v1/agents/delete, /v1/providers, /v1/models/config, workflows, skills, backup.
  • Scheduler: _schedule_owner_check(name) — non-admins may only mutate schedules they own; legacy schedules with an empty user_id stay admin-only.

3.4 Audit log

Append-only audit_log(ts, actor_user_id, actor_username, action, target, details_json, ip), indexed on ts / actor_user_id / action. Written on: login success/failure, user create/update/delete, password reset, team create/update/delete/member-change, permission grant/revoke. Queryable at GET /v1/auth/audit?limit=&actor=&action=&since= (admin-only).

4. Auth flow / login

4.1 Configuration (config.auth.*)

enabled
Master switch. Default false → every request gets SYNTHETIC_ADMIN ({id:"__system__", username:"system", role:"admin"}); single-user mode.
registration_enabled
Default false.
default_role
Default "user".
token_expiry_seconds
Default 86400 (24 h).
JWT secret
Read from config; auto-generated and persisted on first init_auth() if absent.

4.2 Login

POST /v1/auth/login  public
Body {"username","password"} →
  1. AuthDB.authenticate(username, password) — look up by username, reject if disabled, bcrypt.checkpw on the hash.
  2. On success: update_last_login, audit auth.login, generate_token(user) → {"user":{…}, "token":"…"} 200.
  3. On failure: audit auth.login_failed (with IP), {"error":"Invalid credentials"} 401.

4.3 Token

Format
JWT HS256. Payload {user_id, username, role, iat, exp}.
Transport
Authorization: Bearer <token> header. No cookies, no server-side session table — fully stateless.
Verify
verify_token() decodes, checks exp; _get_auth_user() then re-loads the user from auth.db and returns None if the user is missing or disabled.
Refresh
POST /v1/auth/refresh public — verifies the current token, re-loads the user, issues a new token with fresh exp. (Disabled users cannot refresh.)
Logout
Client-side only — discard the token. Tokens self-expire after token_expiry_seconds. There is no revocation list, so a leaked token is valid until expiry (mitigation: short expiry, or disable the user which kills future refreshes but not the in-flight token until it expires).
Public paths
GET /v1/status, GET /v1/auth/me (returns SYNTHETIC_ADMIN or the current user), POST /v1/auth/login, POST /v1/auth/refresh. Everything else flows through _require_auth().

5. Post-login: accessing chats & projects

5.1 Stamping ownership

Sessions are created lazily on the first POST /v1/chat/message (not on model switch / newChat() / PII swap — see "Empty-Session Cleanup"). At creation the handler reads self._auth_user and stamps:

uid = auth_user["id"] if auth_user and auth_user["id"] != "__system__" else ""
session.user_id = uid
ChatDB.update_session_user(session.id, uid)

So under __system__ (auth disabled) sessions are written with an empty user_id — the "legacy / anonymous" bucket that everyone authenticated can later read. When a team chat is started, team_id + visibility='team' are also written.

5.2 Listing chats

GET /v1/sessions → _handle_list_sessions() in handlers/sessions_handler.py:
  1. visible_user_ids = get_visible_user_ids(auth_user) — None for admin (⇒ "all"), else [own_id] + team-members-where-I'm-head.
  2. visible_team_ids = the user's team ids.
  3. ChatDB.list_sessions(visible_user_ids=…, visible_team_ids=…) — SQL filters to user_id ∈ visible_user_ids OR (visibility='team' AND team_id ∈ visible_team_ids) OR user_id = '' (legacy).
  4. Plus the standard hygiene filters: hide 0-message sessions older than 60 s, hide note_chat / hidden-status sessions, etc.

5.3 Opening / mutating a specific chat

Every per-session endpoint (history, rename, delete, settings, memory toggles, …) calls _session_access_check(sid, require_manage=…) first — the ladder is: admin → owner → ownerless-legacy → team-member-can-read / team-head-can-manage → 403. So a team member cannot delete a teammate's private chat, and cannot delete a team chat unless they're the head.

5.4 Projects

GET /v1/agents/<id>/projects → _handle_list_projects() → ProjectManager.list_projects(user_id=…, user_team_ids=…) (skips the user filter entirely for admin). A project is visible if:
  • visibility='user' — caller is owner_user_id or in extra_member_user_ids;
  • visibility='team' — caller is a member of owner_team_id's team;
  • visibility='global' — everyone except those in excluded_user_ids.
Mutating a project (settings, input folders, delete, ingest) requires can_manage_project() ⇒ admin OR owner_user_id only — note even a team head cannot manage a team project they don't own (only the owner or an admin can).

Capability interlock: a user (no allow_projects) is 403'd by the path gate on /v1/agents/<id>/projects* before any ownership logic runs.

5.5 Other per-user surfaces

  • My Schedules — non-admins see only their own; legacy schedules (empty user_id) are admin-only; mutating ops gated by _schedule_owner_check.
  • User profile doc — GET/POST /v1/auth/profile-doc (self), …/update-now (kick the daemon), …/reset (delete the file + the MemPalace mirror drawers).
  • Preferences — POST /v1/auth/me/preferences; password — POST /v1/auth/me/password.

6. MemPalace — per-user memory

MemPalace is imported as an in-process Python package (no MCP, no subprocess). Every drawer carries a wing (namespace) and a room (topic bucket). Per-user memory uses the wing user__<user_id>.

6.1 The user-profile document (auto-maintained memory from chat history)

On disk
agents/main/user_profiles/<uid>.md — six fixed sections (Work context / Personal context / Top of mind / Recent months / Earlier context / Long-term background). Filesystem is the source of truth; gitignored. Versioned into <uid>.history/<ISO>.md (capped 30, kept on reset).
MemPalace mirror
Wing user__<uid>, room user_profile, one drawer per ## section, source_file = user/<uid>#profile/<section_slug>. Updated by purge-then-add (_mirror_user_profile_to_mempalace → _purge_user_profile_drawers then tool_add_drawer per section), so renamed/removed sections don't linger. Only the current profile is mirrored — disk history is not.
Daemon
user-profile daemon polls every ~30 min; per-user gate = daily_summary_enabled AND local hour matches daily_summary_hour_local AND ≥23 h since last run (cooldown in auth.db.user_daily_summary).
Worker
Takes the 100 most-recently-active chats from the last 90 days, runs a delegate with _PROFILE_SYSTEM_PROMPT (never invent / third-person / match the user's language / edit in place / demote stale Top→Recent / no timestamps / 2–6 sentences per section), model picked by _profile_pick_model (refinement → cheapest haiku → cheapest enabled → default). GDPR auto-fallback to a local model if PII is detected in the chat texts. Atomic write via temp + os.replace.
Injection
Round 0 reads <uid>.md (4 KB cap) and prepends [Auto-maintained user profile: …] on the first user message. Stripped before the wire by _ALLOWED_MSG_KEYS. KV-cache invariant: _build_system_prompt stays user-agnostic — per-user content lives only in the first-user-message preamble.

6.2 Chat turns filed into the user's wing

The mempalace-chat-sync daemon (poll ~60 s) mirrors qualifying chat content into whatever wing _resolve_session_wing() returns — for an ordinary owned chat that's user__<uid>. Rooms / source_file conventions:

ContentRoomsource_file
Chat turn (user+assistant pair)chatsession/<sid>#turn/<opening_user_msg_id>
Session summarychat_summarysession/<sid>#summary (content-hashed)
Attachment metadata (filename/mime/size — never bytes)chat_attachmentsession/<sid>…#attach/<mid>/<fname>
Allowlisted tool results (default exa_search, web_fetch, read_document)referencesession/<sid>…#tool/<tool>/<mid>/<idx>

Classifier gate: filing requires the session's save_to_memory mode > 0 (0=off, 1=on, 2=auto). In auto mode each pair is classified (fact/preference/decision/reference filed; generic/refusal/chitchat skipped) — non-streaming, max_tokens:20, fail-open. The per-user memory_chats_default preference overrides the server-wide classifier default. The save_chat_to_memory tool lets the model flip a session to on when the user says "remember this"; per-turn control is via the palace-icon menu (memorise/remove × complete/this/above/below) → memorize_turns / purge_turns.

Cursor: chat_mempalace_sync(session_id PK, last_message_id, last_summary_hash, updated_at) in chats.db. Incognito sessions (status='incognito') are excluded from the sync pass entirely. Session delete purges drawers + closets where source_file LIKE 'session/<sid>%'; archive leaves drawers intact (memory persists).

6.3 Default scoping of mempalace_query

When an agent calls mempalace_query with no explicit wing and no project context, the tool defaults the wing to user__<current_user_id> (read from _thread_local.current_user_id, set per-request by the handler). So an agent acting on behalf of user A naturally searches A's private memory and nothing else.

7. MemPalace — team knowledge

7.1 How it works today

Wing
team__<team_id> — the team_id is the uuid4 hex[:12] from user_teams.id (stable; renaming the team doesn't strand drawers, same-named teams don't collide).
Populated by
Only the mempalace-chat-sync daemon, and only for sessions whose row has visibility='team' AND a non-empty team_id. Same rooms / source_file scheme as §6.2 — the difference is purely which wing the drawers land in. So "team knowledge" = the accumulated content of team-visible chats.
Visibility
A cross-wing mempalace_query (explicit wing=null) returns a team__<tid> drawer only if "team__"+tid ∈ caller's current_team_ids (_thread_local.current_team_ids, set from AuthDB.get_user_teams). Project wings are never returned cross-wing; untyped wings (e.g. brain_code) are treated as shared/everyone.
Session-list parity
The same visible_team_ids that scopes the chat list (§5.2) is what scopes team-memory visibility — consistent boundary.

7.2 What's missing

No team-artifact miner. Projects get a dedicated mempalace-project-sync daemon that mines on-disk input folders + ingested files into project__<id>; scheduled-task artifacts get mined into <agent>_artifacts by the mempalace-miner. Teams get neither — there is no folder or artifact pipeline that feeds team__<tid>. If a team wants a shared knowledge base of documents (not just chat history), today the only path is to create a team-visible project and use its project__<id> wing — which is then governed by project ACLs, not team ACLs. Worth deciding whether that's the intended story or whether a "team input folders" feature should mirror the project one.

8. MemPalace — access control & the wing scheme

8.1 Wing resolution (_resolve_session_wing(info) — server_lib/db.py)

ID-only scheme. Priority, first match wins:

  1. Project-scoped session (session's project field set, resolves to a project id) → project_chat__<project_id>  (note: chat content from a project chat goes to the chat wing project_chat__, kept separate from the knowledge wing project__ so a stale chat answer never out-ranks a source document)
  2. Team-scoped session (visibility='team' + team_id) → team__<team_id>
  3. User-owned session (user_id non-empty) → user__<user_id>
  4. Legacy / anonymous (no owner, not team) → "" → chat-sync skips filing and just advances the cursor (silent — intentional, so the global namespace isn't polluted, but it means anonymous-mode chats are never recoverable from memory)

8.3 The visibility filter (cross-wing search)

When an agent calls mempalace_query with an explicit wing=null (search everywhere), the result set is post-filtered in tool_mempalace_query (brain.py):

def _visible(r):
    w = r["wing"]
    if w.startswith("project__") or w.startswith("project_chat__"):
        return False                       # projects never leak
    if w.startswith("user__"):
        return w == f"user__{current_user_id}"
    if w.startswith("team__"):
        return w in {f"team__{t}" for t in current_team_ids}
    return True                            # untyped wing ⇒ shared

And when an agent is operating inside a project (_thread_local.project set), mempalace_query force-scopes to project__<id> (or project_chat__<id> with include_chat_history) and refuses if the project has no id — it never falls back to a broader wing.

8.5 Project research-mode interaction

Inside a project the system prompt also injects a PROJECT MEMORY block. With research_mode ON it's the strict 3-step flow (mempalace_query → read_document(path=read_path) → answer, REFUSE-on-error, KG-first if kg.enabled) plus a server-side citation validator + synchronous re-round on threshold violation. With it OFF it's a soft "memory is available, use it when relevant". Effective mode = session.research_mode_override if set, else the project default. This is orthogonal to the multi-user wing isolation above but worth noting because it's the other half of "how memory behaves per session".

9. Findings, gaps & recommendations

9.1 Strengths

  • Two-layer authz (path gate + per-resource ownership) means a missed path entry doesn't silently expose a resource — the ownership check is a second net.
  • Single funnel for visibility — get_visible_user_ids() + visible_team_ids drive both the chat list and team-memory visibility, so there's one consistent boundary, not a per-endpoint reimplementation.
  • ID-only wings — using uuid hex[:12] for user/team/project wing keys makes renames and same-name collisions non-issues; projects are structurally incapable of leaking via cross-wing search (hard-coded return False).
  • KV-cache discipline — per-user content (profile, preferences, greeting) is deliberately kept out of _build_system_prompt and injected only in the first-user-message preamble; this is the right call for warm-pool reuse and is documented as an invariant.
  • Audit trail on all identity-affecting operations, with IP capture on login attempts.
  • GDPR interlock with memory — the user-profile worker auto-falls-back to a local model when chat texts contain PII, so the auto-maintained memory pipeline doesn't ship PII to cloud models.

9.2 Gaps / risks

#ItemSeverityNotes
1No JWT revocation listmediumDisabling a user blocks future refreshes but the in-flight token is valid until exp (default 24 h). For a leaked-token scenario the only lever is shortening token_expiry_seconds or rotating the JWT secret (nukes all sessions). Consider a per-user token_epoch in auth.db checked in verify_token.
2Default admin/admin on first runmediumMitigated by a startup warning, but easy to forget. Consider forcing a password change on first admin login when auth.enabled.
3No team-knowledge ingestion pipelinemediumTeams can only accumulate chat-derived memory; document-level shared knowledge has to go through a team-visible project (project ACLs ≠ team ACLs). Decide if "team input folders" should mirror project input folders, or document the project-as-team-KB pattern explicitly.
4Untyped wings are world-readableby designAny wing without a user__/team__/project__ prefix (e.g. brain_code, <agent>_artifacts) is visible to every authenticated user via cross-wing search. Fine for the current shared dev wings, but a footgun if someone hand-creates a wing expecting it to be private — there's no "private custom wing" concept.
5Legacy ownerless sessions are readable by everyone authenticatedby design_session_access_check returns OK for any session with empty user_id. Intentional migration affordance, but it means anything created while auth.enabled=false is shared the moment you flip auth on. A one-shot "assign legacy sessions to admin" migration would close this.
6Team head sees all member resourcesby designget_visible_user_ids gives the head read access to every team member's chats & projects, not just team-visible ones. This is a deliberate "team lead" model but should be surfaced to members (privacy expectation).
7Anonymous-session memory is silently droppedlow_resolve_session_wing → "" → chat-sync advances the cursor without filing. Correct behaviour, just invisible — a log line at debug level would help operators understand why an incognito/anonymous chat "didn't get remembered".
