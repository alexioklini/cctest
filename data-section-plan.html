Brain — "Data" Section

An in-house, Data-Formulator-style AI data-visualization workspace — built as a first-class section like Translation, reusing the existing engine end-to-end. No fork, no second app, no parallel auth.

Reuse: agentic loop · python_exec sandbox · artifacts · GDPR scanner · quota · provider routing · history DB pattern · MemPalace New: DuckDB session store · Vega-Lite renderer · the data_viz tool group (query·transform·verify·compare·anonymise·render·report) · 1 handler mixin · 1 web view Source of inspiration: microsoft/data-formulator (MIT) — concept & prompts vendored, not the codebase
The one-line pitch. The bank's risk / finance / regulatory-reporting teams upload a spreadsheet (or point at a Postgres view), describe a chart in plain language, and Brain writes the pandas/SQL, runs it in the sandbox we already have, and renders an interactive chart — every LLM call passing through the GDPR scanner, quota gate, and provider router that already protect every other Brain feature.

1 · Why build, not embed

Data Formulator ships as pip install data_formulator → uv run data_formulator: a self-contained TS+Python web app on its own port, with its own auth, its own session store, its own LiteLLM config. Three integration options were considered:

OptionEffortVerdict
A — Launch it as a companion app (like the TUI). Brain's launcher.py already launches frontends.LowREJECTED Bank data would flow through its LLM config — bypassing the GDPR scanner, per-user quota, provider routing, audit log. Non-starter for bank data.
B — Port the concept as a new Brain section ("Data"), reusing the agentic loop, sandbox, artifacts, guardrails. Vendor Data Formulator's prompts + chart-semantics module (MIT).MediumCHOSEN The only version where bank data stays inside Brain's safety envelope. Matches how Translation / Workflows / Projects were all built.
C — Wrap their DataAgent.run() as an MCP server.MediumREJECTED The LLM calls still happen inside Data Formulator's process with its own config → re-inherits A's GDPR problem. No benefit.
Decision: Option B. The expensive, distinctive part of Data Formulator's frontend — the drag-and-drop "encoding shelf" — is also the least essential; Brain's chat-first UX replaces it with "describe the chart" + an editable spec preview. The valuable, transferable parts are the prompts and the chart-semantics layer, both MIT-licensed and small enough to vendor with attribution.

2 · What Data Formulator actually is

The core trick

You build a chart by dropping field names onto encoding shelves (x / y / color / size / …). Crucially you can drop fields that don't exist yet ("profit_margin", "rolling_30d_avg"). The AI then writes the pandas/SQL that materializes those columns, runs it in a sandbox, stores the result as Parquet, and renders a Vega-Lite chart whose encodings reference the new columns.

v0.7 (alpha, Mar 2026)

  • Unified DataAgent — observe→think→act loop, actions: visualize / clarify / present, max 5 iterations.
  • 30 chart types via a "semantic chart engine" (workflows/chart_semantics.py).
  • Workspace / data-lake (DuckDB local + Azure Blob), identity-based.
  • Sandboxed exec, code signing, auth, rate limiting — i.e. they re-built half of what Brain already has.
  • LiteLLM → OpenAI / Azure / Anthropic / Ollama.

3 · Target architecture — the reuse map

Everything below the dashed line already exists in Brain. The Data section adds only the four boxed-orange components.

Why a per-session DuckDB file inside the artifact folder

  • Brain already creates agents/<name>/artifacts/<date>_<session_prefix>/ per session and the empty-session cleanup / artifact promotion machinery already manages that folder. Dropping _data.duckdb there means it's cleaned up, scoped, and per-user for free.
  • python_exec's working dir is that artifact folder (CLAUDE.md: "Working dir = artifact session folder"). So generated pandas/SQL can do duckdb.connect('_data.duckdb') with a bare relative path — no path plumbing needed.
  • DuckDB is already a transitive dep (Data Formulator uses it; Brain's project ingest path touches it). One pinned line in requirements.txt.

Relationship to Projects — "a special kind of chat", not a Project

A Data Workbench session is a chat session under the hood. That's why the chat window, the composer, artifact handling, and every guardrail come for free. It borrows the Project pattern — a per-session system-prompt block + an extra tool group — but it does not depend on the Project subsystem. Getting this distinction right is what keeps the change small (and is exactly the scope-creep §12 warns about).

Comes for free (it's a chat session)

  • Chat window & composer — SSE streaming, inline tool-call display, cancellation, compaction, cost logging. The composer reuses the chat composer's model picker + format-aware thinking-level button (per-session sticky, self-correcting on model switch). Dropped: research-mode button (irrelevant), caveman mode (per-model knob, would invalidate warmup KV prefix). The attachment chip is replaced by the Sources tab — same multipart path, different affordance. Added: the Workbench status row (source/table indicator, GDPR-scan badge, ⌘K).
  • Artifacts — the session gets the standard agents/<name>/artifacts/<date>_<sid>/ folder. Anything write_file/python_exec drops there is auto-promoted, versioned, shows in the global Artifacts panel, gets mined. "Pin as Report" is literally writing report.html into that folder and letting _after_file_write() run. The _data.duckdb file lives in that same folder — cleaned up with the session.
  • Guardrails — GDPR pre-flight, quota gate, provider routing, RBAC, audit, RateLimiter — all inside send_message() + the standard handlers. Routing through them is the whole point.

Borrowed as a pattern — NOT a dependency

  • The system-prompt block — a DATA WORKBENCH block injected when _thread_local.data_workbench is set, exactly the way project chats get a PROJECT MEMORY block gated by _thread_local.project. That gating is what keeps the KV prefix byte-stable for normal chats.
  • What it does NOT reuse: ProjectManager / project.json / project input folders / KG extraction / the project-sync daemon / the project__<id> MemPalace wing / research_mode / the citation validator. All of that machinery is irrelevant here and would be dead weight.
  • Mental model: "a chat with one extra system-prompt block, one extra tool group (data_viz), and a DuckDB file in its artifact folder" — much closer to a chat than to a project.
So: chat window — free. Composer (model + thinking) — free, lightly trimmed. Artifacts — free. Guardrails — free. The "project-ness" — borrowed as the gated-prompt-block pattern, not as a coupling to ProjectManager. Conflating it with a real Project would drag in the data-lake / KG / sync-daemon baggage we explicitly don't want.

4 · User experience & mockups

4.1 — Entry point: the sidebar

A new data-view registered exactly like translation-view in web/js/nav.js — same switchView() dispatch, same per-section collapse persistence, same header-star (favourite) plumbing. Sub-tabs mirror Translation's text / document / audio / live pattern.

4.2 — Sources sub-tab

Two ways to get data in. Both feed the same per-session DuckDB.

Upload a file

CSV / TSV / XLSX (multi-sheet) / Parquet / JSON. Goes through the existing multipart upload path used by Translation's document tab and the project ingest — we don't write a new uploader. On arrival: each sheet/table → a DuckDB table; a 20-row sample + schema is computed and shown.

Behind the curtain: a POST /v1/data/sessions/<sid>/upload that calls the same _parse_multipart helper translate.py uses, then duckdb.read_csv / read_parquet / pandas.read_excel → con.register.

Connect a source (Phase C)

Postgres / MSSQL / a project's already-ingested folder. Reuses the connection-string handling Data Formulator's data-loader module formalized — but the credentials live in Brain's existing per-user secret store, and the connection is read-only. The agent sees only the schema + a sample, never raw bulk rows in context.

Bank reality: this is where the GDPR scanner matters most. The sample shown to the model is scanned before it ever reaches the LLM (see §9).

4.3 — History & Reports

1:1 the Translation History pattern (TranslateHistoryDB → DataSessionDB). Every workbench session is a row; "Pin as Report" produces an HTML/PNG artifact in the session's artifact folder — already promoted, already visible in the global Artifacts panel, already mined by the artifact miner. A pinned report is just an artifact with role output.

4.4 — The agentic loop, concretely

This is not a new agent. It's the normal Brain chat loop, in a session whose system prompt has a "DATA WORKBENCH" block (the way project chats get a "PROJECT MEMORY" block) and whose tool set includes code_exec + the data_viz tool group (§7). The DataAgent behavior (plan → maybe clarify → transform → verify → visualise → present) is achieved by vendoring Data Formulator's prompt text into that block, not by writing a state machine.

7 · The data_viz tool family

All of the workbench's data work — querying, transforming, verifying, comparing, anonymising, rendering, reporting — is exposed as discrete tools in one new tool group, data_viz. Same shape as everything else in TOOL_DEFINITIONS: a JSON-schema'd input, a deterministic Python body, a structured return the model observes. The model orchestrates them in the agentic loop; it does not hand-roll pandas in python_exec for these operations (it can still drop to python_exec for genuinely bespoke logic, but the named tools are the paved road — predictable, auditable, individually testable, per Rule 5).

Why named tools, not "the model writes pandas". Every named tool gives us: a stable audit-log line ("ran data_anonymise on table findings, columns ['owner_email','borrower_name'], strategy hash+nullify"), a unit test, a fixed return contract the UI can render, and a place to enforce invariants (anonymise always writes a new table + a mapping file; never mutates in place). Free-form pandas in the sandbox gives us a string of code in a tool_result and nothing structured. For a bank, the structured audit trail is the point.

Added to TOOL_DEFINITIONS in brain.py and the engine mirror (per feedback_brain_tool_duplication — brain.py keeps its own TOOL_DEFINITIONS/TOOL_GROUPS/TOOL_DISPATCH, all three updated). The whole group is enabled only in a data-workbench session's token_config — invisible everywhere else, zero KV-prefix impact on normal chats. All bodies live in engine/tools/data_viz.py; all operate on the per-session _data.duckdb via a bare relative path (cwd = artifact folder, CLAUDE.md invariant).

Two illustrative bodies (the rest follow the same pattern — schema'd input, deterministic body, structured return):

data_verify — assertions are code, not judgement (Rule 5)

# engine/tools/data_viz.py
def tool_data_verify(table: str, checks: list[dict]):
    con = duckdb.connect("_data.duckdb")
    out = []
    for c in checks:
        kind = c["kind"]
        if kind == "sum_equals":
            got = con.sql(f"SELECT sum({c['col']}) FROM {table}").fetchone()[0]
            ok  = abs(got - c["expected"]) <= c.get("tol", 0)
            out.append({"check":c, "passed":ok, "got":got})
        elif kind == "foreign_key":
            bad = con.sql(f"""SELECT * FROM {table} t LEFT JOIN {c['ref_table']} r
                  ON t.{c['col']}=r.{c['ref_col']} WHERE r.{c['ref_col']} IS NULL LIMIT 20""").df()
            out.append({"check":c, "passed":len(bad)==0, "failing_rows":bad.values.tolist()})
        # … not_null / unique / range / regex / in_set / balanced / monotonic / row_count_equals
    return {"passed": all(r["passed"] for r in out), "results": out}

data_anonymise — never mutates, always logs, re-scans

def tool_data_anonymise(table, columns, new_table=None, mapping_out=None):
    new_table   = new_table   or f"{table}_anon"
    mapping_out = mapping_out or f"{table}_anon_map.json"   # in artifact folder
    con = duckdb.connect("_data.duckdb"); df = con.sql(f"SELECT * FROM {table}").df()
    salt, mapping = secrets.token_hex(16), {}
    for spec in columns:
        col, strat = spec["name"], spec["strategy"]
        df[col], m = _apply_strategy(df[col], strat, spec.get("opts",{}), salt)
        if m: mapping[col] = m            # tokenise/hash/fpe → reversible map
    con.register(new_table, df)
    _write_mapping(mapping_out, {"salt":salt, "strategies":columns, "map":mapping})
    residual = pii_scan_dataframe_sample(df)          # the SAME 71-detector scanner
    audit("data_anonymise", table=table, new_table=new_table,
          columns=[c["name"] for c in columns], residual_clean=residual.clean)
    return {"new_table":new_table, "mapping_file":mapping_out,
            "strategy_summary":columns, "residual_scan":residual.to_dict()}
Vendored from Data Formulator (MIT): the data-transform system-prompt text, the DataAgent action-loop instructions, and workflows/chart_semantics.py (the 30-chart-type → Vega-Lite mapping) — into engine/tools/data_viz_prompts.py with a provenance header. The transform/verify/compare/anonymise/report tools are Brain-original (Data Formulator has nothing equivalent — its scope is chart-building, ours adds the bank's data-prep + governance layer). Scope discipline (Rule 2): data_query/data_render_chart ship in Phase A; data_transform lands when free-form pandas proves too unstructured to audit; data_verify/data_compare/data_anonymise/data_report in Phase B, each only built when a real workflow needs it — not all at once on spec.

9 · Bank guardrails — what we get for free

This is the entire reason for Option B. Every one of these is an existing Brain mechanism that the Data section inherits by virtue of routing through send_message() and the standard handlers.

GDPR / PII scanner

Fires at two points: (1) when an uploaded sample is computed — email/ID-bearing columns flagged & masked in the sample shown to the model; (2) round-0 pre-flight in send_message() on the assembled prompt. gdpr_pick_model_for_background-style routing: clean → proceed; findings + a configured local fallback → silent swap to on-prem model; else server_block → refuse. Same 71 detectors, same config (gdpr_scanner), same audit events (pii_detected / pii_auto_fallback / pii_blocked).

Per-user cost quota

QuotaManager pre-flight gate already runs in send_message() round 0. A risk analyst running 40 chart iterations hits the same daily/cycle cap as any other usage. force_local mode swaps to the local model silently; hard_block raises. Local models always bypass — data stays on-prem and free.

Provider routing & fallback

resolve_provider_for_model() is the single source of truth — already used by chat, delegate, scheduler. The Data section picks a model the same way (default Mistral Small for this kind of structured work, per memory; user can switch). Provider down → standard fallback ordering kicks in.

Audit, cost, rate limiting

CostTracker logs every LLM call to costs.db with the authenticated user_id (thread-local already set by the auth layer). RateLimiter sliding window per agent applies. The audit log already records tool calls. Nothing new to build for compliance reporting.

Sandbox isolation

Generated pandas/SQL runs in python_exec — subprocess (sys.executable), timeout-killed, working dir pinned to the artifact folder. We do not hand the model arbitrary shell. DuckDB connection is opened by the generated code inside that sandbox; it can't reach outside the folder.

RBAC

/v1/data/* routes sit behind the same auth the rest of the API uses. A user-role account that can chat can use the workbench; config stays admin-only. Per-user schedules / history visibility rules apply if we later let workbench reports be scheduled.

Net: the bank's security review of the Data section is mostly "it's the chat loop with a chart renderer" — there is no new trust boundary, no new credential store, no new LLM egress path. That is worth far more than the few weeks Option A would have saved.

10 · Phased delivery

A

Phase A — Workbench MVP (text → chart)

  • New data-view in web/js/nav.js + web/js/data.js (Workbench + Sources sub-tabs only).
  • handlers/data_viz.py: POST /sessions, /upload (CSV/XLSX → DuckDB), /tables, /ask (SSE → send_message()).
  • data_viz tool group, Phase-A members: data_query, data_render_chart (and a first cut of data_transform if free-form pandas in python_exec proves too unstructured to audit early); vendored prompt + chart-semantics.
  • SSE events data_chart (+ data_table when a tool materialises a table); vega-embed from CDN in the chart pane; editable spec textarea + client-side re-render.
  • DataSessionDB in server_lib/db.py; DuckDB file in the artifact folder.
  • System-prompt "DATA WORKBENCH" block (vendored DataAgent loop text + the tool-chaining guidance).

Done when: upload risk_register.xlsx → "stacked bar of open findings per entity by tier, top 8" → correct chart renders, GDPR scan visible, cost row logged. Eval: a 10-question canary (5 chart asks, 3 follow-ups, 2 deliberate ambiguities that should trigger clarify) judged against a hand-built gold set — same harness pattern as eval/.

B

Phase B — the data-prep + governance layer (transform · verify · compare · anonymise · report)

  • data_transform (full op vocabulary + lineage), data_verify (assertion engine), data_compare (table diff, 3 modes) — each landed when a real workflow needs it, not all at once (§15).
  • data_anonymise (per-column strategies, mapping-file, re-scan, k-anon warning) + the deliberate anonymisation workflow + the /anonymise and /mapping endpoints (§16.1–16.2).
  • data_report (section assembler → self-contained HTML artifact with provenance + verify panels); the /pin button calls the same body; auto-promoted, shows in global Artifacts, governed by the v8.35 sharing model.
  • History & Reports sub-tab (mirror Translation History UI). /memorize → MemPalace drawer in the user's wing. /render server-side PNG endpoint for edited specs.
  • SSE events for data_verify result panels and data_report artifact-ready.

Done when: a chain — upload two periods → data_transform the join → data_verify ties-out → data_compare the periods → data_render_chart → data_report — produces an HTML artifact carrying the lineage + pass/fail panels; a column flagged by the scanner can be data_anonymised and the re-scan comes back clean; the report opens from History, appears in Artifacts, and the miner has filed the insight.

C

Phase C — Live sources

  • Connect read-only to Postgres / MSSQL (credentials in Brain's existing per-user secret store) or to a project's already-ingested folder.
  • Schema + sample pulled and scanned before anything reaches the model — the GDPR interlock is the hard requirement here.
  • GDPR-triggered auto_anonymise (§16.3): one new enum value in the existing per-category gdpr_scanner config; on a scanner hit, de-identify the named columns, re-scan, proceed on clean data — or fall back to block/local-swap. Built here, not in Phase B, because auto-modifying data on a scanner hit needs the false-positive rate on real bank tables measured first.
  • Optional auto-refresh (cron-style, reusing the scheduler) for a pinned report that should re-run nightly — this is the one place a /schedule follow-up genuinely fits.

Done when: a Postgres view → workbench session → chart, with the connection never returning bulk rows into LLM context, only schema + N-row sample.

D

Phase D — Polish (only if used)

  • Optional lightweight encoding shelf (drag a column chip onto x/y/color) as a shortcut that just composes a prompt — not a parallel codepath.
  • "Recommend a chart" button → the DataRecAgent-style suggestion turn (already mostly there via the suggestions row).
  • Multi-chart dashboards in one report.

Gate on real usage. If the risk team is happy with chat-driven, Phase D may never ship — and that's fine.

14 · Use cases — banking · finance · risk · compliance

Concrete "is this worth it for us" scenarios, framed for a bank's risk / finance / regulatory-reporting / internal-audit / compliance teams. Each is a thing one of those teams does today in Excel + a BI tool + a lot of copy-paste — and which becomes a single chat turn in the Data Workbench, inside Brain's GDPR / quota / audit envelope.

Reading the cards. Each use case lists: WHO the team · ASK what they type · DATA the source file/table they upload · the chart Brain produces · and why on-prem matters for that specific data. The mini-charts are illustrative of the kind of output, not real numbers.

14.1 — Credit-risk portfolio monitoring

WHO Credit Risk / Portfolio Management

ASK "Loan book by rating bucket and segment — stacked bars, flag where exposure in sub-investment-grade grew vs last quarter."

DATA loan_book_q.xlsx (facility-level: borrower, segment, internal rating, EAD, drawn/undrawn, vintage) — joined client-side in DuckDB against last quarter's snapshot.

Output: stacked exposure-by-rating chart + a delta column; follow-ups: "split corporate vs SME", "show only watchlist names", "expected-loss view using these PDs/LGDs".

Why on-prem: facility-level data is borrower-identifying. The scanned sample masks borrower names/IDs before the model sees it; bulk EAD numbers never leave DuckDB-in-the-sandbox.

14.2 — Regulatory capital — RWA / capital-ratio waterfall

WHO Regulatory Reporting / Capital Management

ASK "Waterfall: CET1 ratio from year-end to now — profit, dividend accrual, RWA growth, the IRB model change. Annotate each step in bps."

DATA capital_movements.xlsx (own-funds line items, RWA by approach, period over period) — already an internal artefact for the COREP submission.

Output: a bps waterfall; follow-ups: "do the same for the leverage ratio", "split the RWA move into credit / market / op-risk", "pin this as the figure for the ICAAP narrative".

Why on-prem: pre-publication regulatory figures. This must not touch a third-party LLM endpoint at all — local-model routing or block, per the bank's policy, enforced by the same GDPR config that governs every chat.

14.3 — Liquidity & ALM — LCR composition and maturity ladder

WHO Treasury / ALM

ASK "Contractual maturity ladder: assets vs liabilities by time bucket, plus the cumulative gap line."  then  "now LCR — HQLA Level 1 / 2A / 2B as a stacked bar against net outflows."

DATA maturity_ladder.csv + lcr_components.csv (the same extracts that feed the daily LCR report).

Output: dual-axis bar + cumulative-gap line; a second HQLA-composition chart. Follow-ups: "stress: haircut Level 2B by 25%", "show the 30-day point only", "trend the LCR daily over the quarter".

Why on-prem: intraday liquidity position is highly market-sensitive. Stays in DuckDB inside the sandbox; only the schema + a masked sample is ever in the prompt.

14.4 — Operational-risk & incident analytics

WHO Operational Risk / Internal Control

ASK "Heatmap: Basel event-type × business line, colour by total gross loss this year; ring the single worst cell."

DATA loss_event_register.xlsx (date, event type, business line, gross/net loss, recovery, root cause, status).

Output: the heatmap; follow-ups: "Pareto of root causes", "near-misses only", "monthly loss run-rate vs the op-risk appetite line", "drill into the 'Clients & Products / Markets' cell — list the events".

Why on-prem: loss events often reference named staff, counterparties, legal matters. The scanner masks names/emails in the sample; the register itself stays in the sandbox.

14.5 — AML / financial-crime — alerts & SAR funnel

WHO Financial Crime / AML Operations

ASK "Funnel: transaction-monitoring alerts → triaged → escalated → SAR filed, by month. And a side chart: alert volume by scenario, flag scenarios with a <2% SAR-conversion rate."

DATA tm_alerts.csv (alert id, scenario, generated/closed dates, disposition, SAR ref) + case_outcomes.csv.

Output: stage funnel + a conversion-rate bar with low-yield scenarios highlighted (a classic "is this rule producing only noise?" question). Follow-ups: "median days alert→disposition", "backlog ageing buckets", "which analyst queue is the bottleneck".

Why on-prem: this is SAR-adjacent data — tipping-off-sensitive, names of subjects. Hard requirement that no row reaches an external endpoint; the sample sent to the model is masked, the bulk stays sandboxed. Exactly the case the GDPR/PII routing was built for.

14.6 — Compliance — obligations coverage & control testing

WHO Compliance Monitoring / Assurance

ASK "100% stacked bar: control-test outcomes (effective / deficiency / not-tested) per regulation, sorted by % effective. Then list the overdue tests."

DATA control_testing.xlsx (control id, mapped regulation/obligation, test date, result, owner, remediation status) — the monitoring-plan tracker.

Output: the coverage chart; follow-ups: "obligations with zero mapped controls", "remediation ageing", "trend % effective over the last 4 cycles", "pin as the compliance-committee figure".

Why on-prem: less PII-heavy, but pre-committee assurance findings are confidential. Stays inside Brain; the artifact (pinned report) lands in the user's own artifact space, governed by the sharing model.

14.7 — Finance / FP&A — variance & cost-base analysis

WHO Finance / FP&A / Cost Management

ASK "Actual vs budget by cost centre, year-to-date — sorted by absolute variance, red where over budget >5%. Then a bridge from budget to actual at the total level."

DATA gl_actuals.xlsx + budget.xlsx (cost centre, account, period, amount) — straight off the GL.

Output: ranked variance bar + a budget→actual waterfall. Follow-ups: "split FTE vs non-FTE", "phasing — variance by month", "which accounts drive the IT cost-centre overrun", "forecast full-year run-rate".

Why on-prem: management accounts are MNPI for a listed bank. No external LLM egress — local routing/block per policy.

The decision framing. Every one of these is being done somewhere in the bank right now — in Excel, in a BI tool the data has to be exported to, or by a data analyst on a ticket queue. The Data Workbench doesn't add a capability the bank lacks; it moves that work inside the perimeter you already trust — the same GDPR scanner, quota gate, audit log, sandbox, and (with the v8.35 sharing model) the same private-by-default artifact space. The build cost is the data_viz tool group + one handler + one web view + a DuckDB-per-session file — and zero core-path changes. If even two of these seven teams adopt it, that's the BI-tool data-export step removed for their sensitive datasets — which is the part the security review actually cares about.
Where it is not worth it (be honest). Pixel-perfect, recurring board-pack visuals with house styling → keep the BI tool. Real-time dashboards over a streaming source → out of scope (this is ask-driven, not a live tile). 50-page COREP/FINREP templated returns → that's a reporting engine, not a chart workbench. The Data Workbench's sweet spot is the ad-hoc, exploratory, "I need to see this cut of the data now, on data too sensitive to export" question — which, for risk/compliance/finance, is a large fraction of the actual day.

15 · Beyond charts — transform · verify · compare · report

Charts are the demo; for a bank the day-job is the un-glamorous middle: get the data into shape, prove the numbers tie out, see what moved since last time, and hand someone a documented artefact. Each of those is a data_viz tool from §7, and a workbench turn chains them. Below: the workflow shape, then concrete banking chains.

15.1 — Data manipulation & transformation

The data_transform tool is the workhorse: a declarative op-pipeline (filter/derive/aggregate/join/pivot/window/bucket/rename/cast) that compiles to one DuckDB statement, always writes a new table, and returns a lineage blob. Examples in our world:

  • Credit risk: "join the loan book to the rating master, derive ead_drawn = limit − available, bucket vintage into FY-cohorts, aggregate EAD by cohort × rating." → one transform call, one new table loan_by_cohort_rating, lineage shows the join keys + the derived-column formula.
  • Finance: "unpivot the budget sheet from wide-by-month to long, cast period to date, join to actuals on (cost_centre, account, period), derive variance and variance_pct."
  • Treasury: "bucket cash flows into the regulatory time bands, sign assets +/liabilities −, window-sum for the cumulative gap."
  • AML: "derive days_to_disposition, bucket into ageing bands, aggregate alert count + SAR count by scenario × month, derive sar_rate."

When a transform is genuinely beyond the op vocabulary (a gnarly recursive CTE, a stats library call), the model drops to python_exec — but then the structured-lineage benefit is lost, so the system prompt nudges it to prefer data_transform and only escape when it must. (Rule 2: don't grow the op vocabulary speculatively — add an op when a real workflow needs it.)

15.2 — Verification & reconciliation

data_verify runs assertions as pure code — no LLM judgement (Rule 5). The banking flavour is reconciliation and data-quality gating:

  • Reg reporting: "verify: sum of rwa in the transformed table equals the COREP control total ±€1k; every entity_id exists in the legal-entity master; cet1_ratio is between 8% and 30% (sanity)." → a pass/fail panel; on fail, a sample of the offending rows so the analyst sees which entity broke the FK.
  • Finance: "verify the GL extract balances: debits == credits within each journal; row count matches the trial-balance line count; no null cost-centres." — a gate before any chart is even drawn.
  • Credit risk: "verify EAD is never negative; PD ∈ (0,1); every facility has a rating; total EAD ties to the balance-sheet number ±0.5%."
  • Op risk / AML: "verify no loss event has gross < net; every SAR ref in case_outcomes appears in tm_alerts (no orphan SARs); dates are monotonic within a case."

A failed data_verify can be wired as a hard stop in the workflow ("don't render the chart, don't write the report — the numbers don't tie out") — the model gets the failing rows and either fixes the transform or tells the user the source data is broken. That's the difference between "AI made a pretty chart" and "the figure in this report was checked."

15.3 — Comparison

data_compare is the "what moved?" tool — diff two tables on keys, in three modes (rows set-diff / values field-level deltas / aggregate group-total deltas):

  • Period-over-period: this quarter's loan book vs last quarter's — added facilities, repaid facilities, rating migrations (the values mode on the rating column), net EAD delta by segment. Feeds the credit-risk QoQ commentary directly.
  • Actual vs plan: aggregate mode on (cost_centre) → the variance table, already broken into the bridge components.
  • Source vs reconciled: the raw extract vs the post-adjustments table — exactly which rows the manual adjustments touched, and by how much. An auditor's dream: the diff is the adjustment journal.
  • Two model runs / two scenarios: base-case RWA vs stressed RWA, per portfolio — the stress impact, line by line.
  • Snapshot drift: last month's control-testing tracker vs this month's — which controls flipped from effective→deficient.

15.4 — Report generation

data_report assembles an ordered list of sections — headings, model-written narrative, inline charts, row-capped tables, KPI tiles, data_verify result panels, and an auto-filled provenance section (source files + their scan status, the transform lineage, the model + timestamp, the verify outcomes) — into one self-contained HTML file in the session's artifact folder. Because it lands in that folder it's already promoted, versioned, mined, and governed by the v8.35 sharing model (private by default; share to users/team/global explicitly). It is the "Pin as Report" output — the UI button and the tool call hit the same body.

  • Credit-risk quarterly pack: EAD-by-rating chart, the QoQ-migration table from data_compare, a KPI tile for total EAD + the delta, a verify-panel showing "ties to balance sheet ✓", a narrative paragraph the model drafted, provenance. One artefact, reproducible, auditable.
  • ICAAP capital figure: the bps waterfall + a one-paragraph explanation of each step + the verify-panel ("RWA ties to COREP control total ✓") → pinned as the figure the narrative references.
  • AML MI: the alert→SAR funnel, the low-yield-scenario flag table, backlog-ageing KPIs, the verify-panel ("no orphan SARs ✓").
  • Reconciliation evidence: the data_compare source-vs-reconciled diff + the verify results + provenance → an audit-evidence artefact, not a screenshot.
The "verify-panel in the report" detail matters. A bank report whose figures are accompanied by the machine-checked assertions that produced them — and whose provenance section names the source files, their PII-scan status, and the transformation lineage — is qualitatively different from a chart pasted into a deck. That's the bit that makes this defensible in front of internal audit, and it falls out of the tool design for free.

16 · Automatic anonymisation — workflow & GDPR-triggered redaction

Two distinct things, both built on the data_anonymise tool: (a) a deliberate "anonymise this before it leaves the bank" workflow the user or agent invokes; (b) an automatic path where the GDPR scanner detecting PII offers — or, in strict mode, performs — redaction instead of just blocking. For a bank, "the scanner said no" is often less useful than "the scanner anonymised the two offending columns and let the safe analysis through." This section is the design for both.

16.1 — Strategies (what data_anonymise can do per column)

StrategyEffectReversible?Banking example
hashSalted SHA-256; optional keep_prefix (e.g. first 3 chars of an IBAN's country+bank code)No (salt discarded) / Yes (salt kept in mapping file)Borrower ID → stable pseudonym; same borrower hashes the same within the run
tokeniseReplace with consistent surrogate IDs (CUST_00001…); the real↔surrogate map written to the mapping fileYes — via the mapping file (RBAC'd, audited download)Customer name/account → surrogate, so a downstream analyst can still group by "customer" without knowing who
redactRegex-replace PII inside a free-text column (emails, IBANs, phone numbers, national IDs) with [REDACTED:email] tags — reuses the 71-detector regex setNoOp-risk "root cause notes" or AML "narrative" columns — keep the prose, scrub the identifiers
generaliseCoarsen: date→month/quarter, age→band, postcode→district, salary/amount→bucket, exact timestamp→hourNo (lossy by design)HR-adjacent risk data: keep "Q2 2026 / age 30-39 / band C" instead of exact DOB + salary
nullifyDrop the column's values entirelyNoA column that's PII and not needed for the analysis at all
shufflePermute a single column's values across rows — breaks the row-link for that attribute while keeping its marginal distributionNoAnonymising a demo/test dataset where one sensitive attribute must stay "realistic" but un-linkable
noiseAdd ε-bounded numeric jitterNoAggregate figures shared externally where exact numbers are MNPI but the shape is fine
fpeFormat-preserving encryption — output keeps the input's shape (16-digit stays 16-digit)Yes — with the key (kept in mapping file)Card PANs / account numbers that must remain syntactically valid for a downstream system to ingest the test extract

Hard invariants of the tool (enforced in code, not prompt): never mutates the source table — always writes <table>_anon; if any strategy is reversible, the salt/key/map goes to a mapping file inside the artifact folder (per-session-scoped, RBAC'd to the session owner, auto-cleaned with the session, downloadable only via an endpoint that logs the download as an audit event); re-runs the full GDPR scanner on the output and returns residual_scan — if it's not clean, the tool says exactly what still leaked so the model can treat another column. k-anonymity warning: if generalising leaves quasi-identifier combos with group size < k (default 5), the tool flags it (it doesn't silently claim "anonymous").

16.3 — Workflow B — GDPR-scanner-triggered automatic anonymisation

Today the GDPR scanner's outcomes (per the existing gdpr_scanner config: ignore / warn / block, plus the default_local_fallback_model swap) are about routing the request. 16.4 — Where the bank tunes it

  • Per-category policy (existing gdpr_scanner config, new value): e.g. email: auto_anonymise, national_id: block, name: warn. Same shape the bank already configures; no new settings page.
  • Default strategy per detector category (new, small): email→redact, IBAN→hash-keep-country-prefix, customer-name→tokenise, DOB→generalise-to-month, etc. Sensible defaults shipped; the bank can override. The agent can also propose a different strategy in the deliberate workflow — the auto path uses the default.
  • Keep-or-drop the original after auto-anonymise at ingest: config flag. Default = drop (safest); some teams may need the original retained under tighter RBAC for a reconciliation step.
  • k threshold for the k-anonymity warning on generalisation (default 5).
  • Mapping-file retention: per-session-auto-clean (default) vs. pin-on-request. Downloading one is always an audit event.
Scope honesty (Rule 2). The deliberate workflow + the data_anonymise tool ship together in Phase B. The GDPR-triggered auto path is a Phase B/C addition — it's the higher-value but higher-risk piece (auto-modifying data on a scanner hit needs careful eval: false-positive PII detections would auto-mangle good columns). Build the manual tool first, watch how the scanner behaves on real bank tables, then turn on auto_anonymise per category once the false-positive rate on those tables is known. Don't ship the auto path on day one because it sounds neat.
What this is not. Not formal differential privacy (the noise strategy is ε-bounded jitter, not a DP mechanism with a privacy budget across queries). Not a guarantee of re-identification-impossibility (k-anonymity is a heuristic; we flag low-k, we don't certify anonymity). Not a substitute for a data-sharing agreement and a DPIA — it's the technical control that makes those easier to satisfy, not a replacement for the governance process. The honest claim: it removes the identifiers the bank's own scanner can detect, logs exactly what it did, and re-checks its own work — which for "share this extract internally / with an auditor under NDA" is the bar that matters, and which beats the status-quo (someone hand-deleting columns in Excel with no log).
